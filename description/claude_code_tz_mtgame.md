## 3. Итоговая спецификация (ТЗ для реализации)

### 3.1 Какой функционал реализуется
1. Веб-интерфейс live-сбора волейбольной статистики по матчу (сет/матч/розыгрыш/игрок).
2. Подготовка матча перед стартом: заявки команд, стартовые расстановки, цвета команд, настройки правил.
3. Оперативная фиксация игровых действий через быстрые действия и полное меню действия игрока.
4. Централизованная лента событий розыгрышей с возможностью редактирования/удаления.
5. Автоматические правила расчета отдельных исходов (эйс, переходы, подсказки по либеро).
6. Поддержка завершения сета, старта следующего сета с копированием расстановки, завершения матча.
7. Пост-матч функционал: выбор MVP двух команд, повторное открытие матча для корректировок, отображение карточки матча.

### 3.2 Состав компонентов/файлов (рекомендуемая декомпозиция)
1. `LeaguePage` или аналог: список лиг/турниров, поиск, вход в матч.
2. `TournamentSettingsPage`: правила турнира и матча.
3. `MatchCardListPage`: выбор конкретной игры.
4. `RosterModal`: выбор игроков в заявку команды, сохранение.
5. `LineupModal`: выбор стартовой расстановки по зонам.
6. `StatsCollectionPage`: основной экран сбора статистики.
7. `TeamActionMenu`: `Заявка`, `Либеро`, `Таймаут`, `Цвет`, `Техпоражение`, `+1 очко`.
8. `BottomControlBar`: служебное меню, смена сторон, звуковой сигнал.
9. `PlayerQuickActions`: быстрые действия на иконке игрока.
10. `PlayerFullActionModal`: полный список действий по игроку.
11. `EventTimeline`: вертикальная лента событий розыгрышей.
12. `EventEditModal`: изменение автора/типа события, удаление события.
13. `GameSettingsModal`: настройки текущего матча, действия табло/виджета.
14. `SetFinishFlow`: кнопка завершения сета и старт следующего сета.
15. `MatchFinishFlow`: подтверждение завершения игры, экран завершенной игры.
16. `MvpModal`: выбор MVP по обеим командам.
17. `PublicMatchCardPage`: итоговая публичная статистика и лучшие игроки.
18. `domain/match-engine` (или аналог): бизнес-логика состояния матча и автоправил.
19. `store/match-state` (или аналог): реактивное состояние матча.
20. `api/match` (или аналог): чтение/запись матчевых данных и событий.

### 3.3 Логика работы (функциональные правила)
1. До старта матча должны быть выбраны заявки обеих команд.
2. До старта розыгрышей должна быть сохранена стартовая расстановка по зонам.
3. На старте розыгрыша у подающего доступны быстрые действия `Подача` и `Ошибка`.
4. После фиксации подачи система требует указать прием у соперника.
5. Если отмечена ошибка приема, подаче автоматически присваивается эйс и начисляется очко.
6. Если после приема не выбрано промежуточное действие и зафиксирована атака партнера, прием трактуется как качественный/отличный.
7. После приема должны быть доступны быстрые действия для продолжения розыгрыша (`Прием`, `Атака`, `Блок`) в соответствии с текущим контекстом.
8. Ошибка на подаче автоматически завершает розыгрыш, начисляет очко сопернику, выполняет переход.
9. Для либеро действует отдельная логика замены: разрешенные позиции, подсветка кандидатов, авто-подсказка на замену при выходе либеро в переднюю линию.
10. Все действия пишутся в хронологическую ленту с привязкой к розыгрышу и счету.
11. У каждого события в ленте есть флаг хайлайта (звездочка).
12. Любое событие можно открыть на редактирование и изменить автора/тип.
13. Любое ошибочно внесенное событие можно удалить.
14. При достижении порога очков сета появляется `Завершить сет`.
15. Новый сет требует подтверждения расстановки; допустим выбор по образцу предыдущего сета.
16. При достижении победы в матче появляется `Завершить игру`.
17. После завершения матча доступен выбор MVP по обеим командам.
18. После завершения матч можно переоткрыть и скорректировать данные.
19. Итоговая карточка матча должна показывать агрегированную статистику и выбранных лучших игроков.

### 3.4 Используемые технологии/библиотеки (что видно и что требуется)
1. По видео видно SPA-веб интерфейс (темная тема), но конкретные библиотеки не идентифицируются.
2. Реализацию делать в текущем стеке проекта (React/Vue/Angular/другое) без привязки к конкретному фреймворку из видео.
3. Нужен устойчивый стейт-менеджмент для live-изменений (локальный store + синхронизация с API).
4. Нужен слой доменной логики (матч-движок), не смешанный с UI.
5. Нужна серверная персистентность событий и состояния матча.
6. Для режима хайлайтов нужен флаг события `isHighlight` и экспорт списка помеченных эпизодов.
7. Для табло/виджета нужен endpoint/ссылка и обновление по кнопке.

### 3.6 Пошаговый план реализации
1. Реализовать доменную модель матча, сета, розыгрыша, события, замены и MVP.
2. Реализовать API для чтения/записи состояния матча и событийной ленты.
3. Реализовать экран подготовки матча: заявки, расстановка, цвета, правила.
4. Реализовать основной экран сбора: поле команд, верхний счет, лента событий, нижние контролы.
5. Реализовать быстрые действия на иконках игроков и контекстные переходы между фазами розыгрыша.
6. Реализовать полное меню действий игрока.
7. Реализовать автоправила: эйс/ошибка приема/переход/ограничения и подсказки либеро.
8. Реализовать редактирование и удаление событий через карандаш в ленте.
9. Реализовать завершение сета и старт следующего с копированием расстановки.
10. Реализовать завершение матча, экран завершенной игры и кнопки пост-матча.
11. Реализовать модалку MVP на 2 команды и сохранение выбора.
12. Реализовать итоговую карточку матча с агрегированными показателями и лучшими игроками.
13. Добавить тесты доменной логики (автоправила, переходы, замены либеро, пересчет счета).
14. Добавить e2e-сценарии для полного прохода матча от старта до завершения.

### 3.7 Критерии приемки
1. Пользователь может полностью провести матч от заявки до завершения без ручного редактирования базы.
2. Счет по очкам и сетам обновляется автоматически и корректно.
3. Лента событий сохраняет хронологию и поддерживает правки/удаления.
4. Логика либеро соответствует описанному сценарию с подсказками и корректными заменами.
5. Кнопки `Завершить сет` и `Завершить игру` появляются только в корректных условиях.
6. MVP можно выбрать для обеих команд после завершения игры.
7. Переоткрытие завершенного матча позволяет внести изменения и пересчитать агрегаты.
8. Карточка матча показывает итоговую статистику и лучших игроков.
