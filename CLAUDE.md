# VolleyStats PWA

PWA-приложение для ведения волейбольной статистики в реальном времени. Заменяет ручной ввод в Excel/MTGame. Язык интерфейса — только русский. Общение с пользователем по проекту — на русском языке.

## Быстрый старт

```bash
pnpm install
pnpm dev          # dev-сервер на http://localhost:4100
pnpm build        # production сборка (apps/web/dist/)
```

## Архитектура ввода матча (Rally-based)

Ввод матча построен на **розыгрышах (Rally)**. Каждый розыгрыш проходит через конечный автомат (FSM):

```
IDLE → [Начать розыгрыш] → SERVE → RECEPTION → IN_PLAY → RALLY_OVER → IDLE
```

### Ключевые принципы

1. **Отслеживаем обе команды** — домашняя команда с реальными игроками, соперник с генерическими (Игрок 1..6). `awayLineup` содержит `away-player-1`..`away-player-6` (см. `lib/awayPlayers.ts`).
2. **Горизонтальная площадка** — зоны отображаются в 3 ряда × 2 колонки (как в MTGame):
   ```
   [5] [4]           [2] [1]
   [6] [3]  → Сетка  [3] [6]   (away зеркально)
   [1] [2]           [4] [5]
   ```
   Левая колонка = задняя линия, правая = передняя (у сетки). Константы `HORIZONTAL_ROWS` и `AWAY_HORIZONTAL_ROWS` в `DualCourtView.tsx`.
3. **Адаптивный FSM**:
   - **Соперник подаёт** → 2 кнопки: «В игру» (neutral) и «Ошиб» (error). Качество подачи (ace/pressure/in_play) определяется **ретроактивно** по приёму нашей команды.
   - **Мы подаём** → 4 кнопки: Эйс, Услож, В игру, Ошиб (как и раньше).
   - **Мы подаём нейтрально** → переход в `reception` (соперник принимает)
4. **Ретроактивное обновление подачи соперника** — при регистрации приёма `processAction` обновляет quality serve-события соперника: `++`/`+` → in_play, `!`/`−`/`/`/`=` → pressure, ошибка приёма → ace. Обновлённые события возвращаются через `outcome.updatedEvents` и сохраняются в БД через `db.matchEvents.put()`. Старые `autoGenerated` ace-события больше НЕ создаются (но в старых данных могут остаться — фильтруются через `!e.meta?.autoGenerated`).
5. **Ротация** — автоматическая при переходе подачи (side-out). `rotateLineup()` в `matchEngine.ts`.
6. **Либеро** — ручное управление через `LiberoControl` + автозамена при ротации:
   - Поддерживается до 2 либеро на матч (`homeLiberoIds[]`, `homeLiberoReplacements{}`)
   - **Только один либеро на площадке** одновременно. Кнопка второго заблокирована пока первый играет.
   - Не может стоять на передней линии (зоны 2, 3, 4)
   - Не может подавать (зона 1 при своей подаче)
   - `getLiberoEligiblePlayers(lineup, isTeamServing)` — зона 1 доступна только при подаче соперника
   - `autoSwapLiberoOut()` срабатывает после ротации, показывает toast-уведомление
7. **Зонно-зависимые действия** — кнопки в попоевере зависят от зоны игрока (передняя/задняя линия). Передняя линия (зоны 2,3,4) получает дополнительные блок-опции (смягчение, касание).
8. **Замены игроков** — через `SubstitutionModal` в `SideMenu`. Макс 6 замен на сет (`VOLLEYBALL_RULES.MAX_SUBSTITUTIONS_PER_SET`). Правило обратной замены: если А заменён на Б, то А может вернуться только за Б. Либеро-замены не считаются. Счётчик сбрасывается при новом сете.

### Поток данных розыгрыша

```
live.tsx → beginRally() → matchStore.ts → startRally() [matchEngine.ts]
                                              ↓
live.tsx → executeRallyAction() → matchStore.ts → processAction() [matchEngine.ts]
                                              ↓
                                    outcome = { newEvents, updatedEvents, rallyComplete, sideOut }
                                    ├── persist newEvents (db.matchEvents.add)
                                    ├── persist updatedEvents (db.matchEvents.put) ← ретроактивное обновление подачи
                                    └── outcome.rallyComplete?
                                        ├── false → update rally state, continue
                                        └── true → buildRallyRecord(), persist to DB
                                                  ├── side-out? → rotateLineup(), autoSwapLiberoOut()
                                                  └── save rally + events to IndexedDB

live.tsx → handleUndo() → currentRally? undoLastRallyAction() : undoLastRally()
                                                                     ↓
                                              delete rally + events, restore score,
                                              reverse rotation + undo auto-libero swaps
```

- `layout.tsx` — нижняя навигация (Главная, Матчи, Команды, Статистика).
- `routes/dashboard.tsx` — главная страница.

- `routes/teams/[teamId].tsx` — управление командой (CRUD игроков).
- `routes/matches/new.tsx` — настройка матча: пошаговый флоу (1. выбор команд → 2. дата/время матча (`datetime-local` + кнопка «Сейчас») → 3. кто подаёт первым → 4. расстановка домашней команды + выбор либеро). Подача обязательна перед расстановкой. **Без расстановки соперника.**
- `routes/matches/[matchId]/live.tsx` — лайв-ввод матча. Содержит FSM розыгрыша, DualCourtView, ActionPanel, LiberoControl, EventFeed, SubstitutionModal, SideMenu (таймауты + замены).
- `routes/matches/[matchId]/stats.tsx` — детальная статистика матча с секциями (Общее, Подача, Приём с разбивкой ++/+/нейтр/ошиб, Атака, Блок, Защита) + Excel-экспорт. Фильтрует `autoGenerated` события.

- `routes/matches/index.tsx` — история матчей.
- `routes/players/[playerId].tsx` — карточка игрока.

### Компоненты лайв-матча (apps/web/src/components/live/)
- `DualCourtView.tsx` — горизонтальная площадка: домашняя команда (3×2) | сетка | соперник (3×2 зеркально). `ActionPopover` — всплывающие кнопки на кружке игрока с зонно-зависимыми действиями.
- `ScoreBar.tsx` — счёт, номер сета, индикатор фазы, таймауты/замены + **лайв-статистика** (grid 3-колоночный: домашние | категория | гости). Подача: `8 1э 2у 1о` (всего, эйсы, усложнённых, ошибки). Приём: `3/5 60%` (хороших/всего %). Получает `events` prop для текущего сета.
- `EventFeed.tsx` — вертикальный таймлайн в стиле v3 (glass-карточки, score pins, home слева / away справа). Отображает `quality`-метки ошибок (Ош.подачи, Ош.атаки, Уд.в блок и т.д.).
- `SetupLineupModal.tsx` — модалка расстановки для нового сета (только домашняя команда, горизонтальная раскладка).

### Доменная логика
- `lib/matchEngine.ts` — **чистая доменная логика** (~700 строк):
  - `startRally()` — инициализация розыгрыша, определение стартовой фазы
  - `processAction()` — FSM переходы, подсчёт очков, ретроактивное обновление подачи. Возвращает `ActionOutcome { newPhase, rallyComplete, sideOut, newEvents, updatedEvents }`
  - `buildRallyRecord()` — создание записи Rally из завершённого розыгрыша. `wasAce` определяется по `meta.quality === 'ace'`
  - `rotateLineup()` / `reverseRotateLineup()` — ротация зон при переходе подачи / обратная ротация для undo
  - `autoSwapLiberoOut()` — автозамена либеро при попадании на переднюю линию или зону подачи
  - `getScoreChange()`, `isPointForTeam()`, `isPointForOpponent()` — логика очков. `defense` в `isPointForOpponent`
  - `getLiberoEligiblePlayers(lineup, isTeamServing)` — доступные для замены либеро (зона 1 только при подаче соперника)
  - `getAvailableActions(phase, servingTeamId, receivingTeamId, zone?)` — доступные действия по фазе, зонно-зависимые
  - `getAwayAvailableActions(phase, servingTeamId, zone?)` — действия для соперника (подача: 2 кнопки «В игру»/«Ошиб»; остальные фазы — без изменений)
  - `isFrontRow()` — проверка передней линии (зоны 2, 3, 4)
  - `isSetWon()`, `countSetsWon()`, `isMatchWon()` — логика победы

### Данные
- `db/database.ts` — Dexie.js v2: таблицы `teams`, `matches`, `matchEvents`, `specialEvents`, `rallies`.
- `db/seeds.ts` — seed-данные (INEX TEAM vs Комус, 10 + 6 игроков, тестовый матч).
- `db/sync.ts` — фоновая синхронизация IndexedDB ↔ сервер (pushToServer, pullFromServer, startBackgroundSync). `pullFromServer` мержит по `updatedAt` (teams, matches — кто новее, тот и остаётся; events/rallies — immutable, bulkPut). Включает `rallies` в sync payload.
- `stores/matchStore.ts` — Zustand store: `beginRally`, `executeRallyAction`, `cancelRally`, `undoLastRallyAction`, `undoLastRally`, `loadMatch`, `updateMatch`, `addSpecialEvent`. Управляет `currentRally: RallyState`. Содержит `notification: string | null` + `clearNotification()` для toast-уведомлений. `executeRallyAction` обрабатывает `outcome.updatedEvents` (ретроактивное обновление подачи) через `db.matchEvents.put()`. `undoLastRally` — откат завершённого розыгрыша: удаление rally+events, восстановление счёта, обратная ротация (`reverseRotateLineup`), откат авто-замен либеро. После ротации вызывает `autoSwapLiberoOut()`.
- `stores/teamStore.ts` — Zustand store команд.

### Утилиты
- `lib/utils.ts` — `cn()` (clsx + tailwind-merge).
- `lib/awayPlayers.ts` — генерические ID для соперника (`away-player-1`..`away-player-6`, `away-libero-1`..`away-libero-2`), генерация `awayLineup`, display-имена.
- `lib/exportProtocol.ts` — Excel-экспорт протокола матча. Фильтрует `autoGenerated` события. Эйсы считаются по `meta.quality === 'ace'`. Разделяет ошибки атаки на `attackErrors` (невынужденные) и `attackBlocked` (удар в блок).

## Модель данных

### Match
```typescript
{
  id, date, homeTeamId, awayTeamId,
  sets: MatchSet[],          // массив сетов с очками
  currentSet: number,
  status: 'live' | 'finished',
  homeLineup: LineupEntry[], // текущая расстановка домашней команды (6 записей)
  awayLineup: LineupEntry[], // ВСЕГДА пустой [] — соперника не отслеживаем
  setLineups?: SetLineup[],  // расстановки по сетам
  servingTeamId?: string,    // кто сейчас подаёт
  firstServeTeamId?: string, // кто подавал первым (для чередования по сетам)
  homeLiberoId?: string,     // ID первого либеро (legacy)
  homeLiberoIds?: string[],  // до 2 либеро домашней команды
  homeLiberoReplacements?: Record<string, string>, // liberoId → replacedPlayerId
  homeLiberoReplacedPlayerId?: string, // legacy: кого первый либеро заменил
  homeTimeouts, awayTimeouts, homeSubstitutions, awaySubstitutions,
  createdAt, updatedAt
}
```

### MatchEvent
```typescript
{
  id, matchId, rallyId?, setNumber, timestamp,
  teamId, playerId,        // 'opponent-error' для ошибки соперника
  action: ActionType,      // 'serve' | 'attack' | 'block' | 'reception' | 'defense'
  result: ActionResult,    // 'success' | 'error' | 'neutral'
  scoreHome, scoreAway,    // счёт ПОСЛЕ этого события
  meta?: { quality?, autoGenerated? }  // quality: ace, pressure, in_play, serve_error, excellent, positive, ok, negative, over, half, reception_error, attack_error, attack_blocked, block_error, soft, touch, defense_error, setting_error
}
```

### Rally
```typescript
{
  id, matchId, setNumber, rallyNumber,
  servingTeamId, serverPlayerId,
  scoreHomeBefore, scoreAwayBefore, scoreHomeAfter, scoreAwayAfter,
  pointWonByTeamId?,
  wasAce, wasServeError, wasReceptionError,
  timestamp
}
```

### SpecialEvent
```typescript
{
  id, matchId, setNumber, timestamp, teamId,
  type: 'substitution' | 'timeout' | 'libero_in' | 'libero_out' | 'set_start' | 'set_end',
  meta?: { playerIn?, playerOut?, description? }
}
```

### Правила начисления очков
- **+1 нашей команде**: `result === 'success'` + (`action` = attack | block | serve)
- **+1 сопернику**: `result === 'error'` + (`action` = serve | attack | block | reception | defense)
- Приём success/neutral — **не даёт очков**
- Защита success/neutral — **не даёт очков** (мяч поднят / в игре)
- Волейбольные правила: 25 очков/сет, 15 тайбрейк, 2 очка преимущества, 3 сета для победы

### Расширенные оценки (quality)

Оценки хранятся в `MatchEvent.meta.quality`. Поле `result` управляет очками, `quality` — для аналитики и отображения ошибок.

**Подача домашней команды** (4 кнопки):
| UI | result | quality | Очко |
|----|--------|---------|------|
| Эйс | success | `ace` | +1 нам |
| Услож | neutral | `pressure` | нет |
| В игру | neutral | `in_play` | нет |
| Ошиб | error | `serve_error` | +1 сопернику |

**Подача соперника** (2 кнопки + ретроактивное обновление quality):
| UI | result | quality | Очко |
|----|--------|---------|------|
| В игру | neutral | `in_play` (обновляется ретроактивно по приёму) | нет |
| Ошиб | error | `serve_error` | +1 нам |

Ретроактивное обновление quality подачи соперника по результату приёма:
| Приём | quality подачи |
|-------|---------------|
| `++` (`excellent`) или `+` (`positive`) | `in_play` |
| `!` (`ok`), `−` (`negative`), `/` (`over`), `=` (`half`) | `pressure` |
| Ошибка приёма (`reception_error`) | `ace` (→ +1 сопернику через reception error) |

**Приём** (домашняя 7 кнопок / соперник 3 кнопки: +, 0, Ошиб):
| UI | result | quality | Очко |
|----|--------|---------|------|
| ++ | success | `excellent` | нет |
| + | success | `positive` | нет |
| ! | neutral | `ok` | нет |
| − | neutral | `negative` | нет |
| / | neutral | `over` | нет |
| = | neutral | `half` | нет |
| Ошиб | error | `reception_error` | +1 сопернику |

**Атака in_play** (4 кнопки):
| UI | result | quality | Очко |
|----|--------|---------|------|
| + | success | — | +1 атакующей |
| − | error | `attack_error` | +1 сопернику (невынужд: сетка, аут, антенна) |
| Б | error | `attack_blocked` | +1 сопернику (удар в блок) |
| 0 | neutral | — | нет (мяч в игре) |

**Блок in_play** (2–4 кнопки, зависит от зоны):
| UI | result | quality | Очко |
|----|--------|---------|------|
| + | success | — | +1 блокирующей |
| − | error | `block_error` | +1 сопернику |
| с | neutral | `soft` | нет (смягчение, мяч на своей стороне) |
| 0 | neutral | `touch` | нет (касание, мяч на стороне соперника) |

**Защита in_play** (2 кнопки):
| UI | result | quality | Очко |
|----|--------|---------|------|
| + | success | — | нет (мяч поднят) |
| − | error | `defense_error` | +1 сопернику |

**Передача in_play** (1 кнопка):
| UI | result | quality | Очко |
|----|--------|---------|------|
| − | error | `setting_error` | +1 сопернику |

### Отображение ошибок

Quality-метки ошибок в EventFeed и Timeline:
- `serve_error` → "Ош.подачи"
- `reception_error` → "Ош.приёма"
- `attack_error` → "Ош.атаки"
- `attack_blocked` → "Уд.в блок"
- `block_error` → "Ош.блока"
- `defense_error` → "Ош.защиты"
- `setting_error` → "Ош.передачи"

### Excel-экспорт (exportProtocol.ts)

Колонки: № | ФИО | Очки | Ошибки | Подача(Всего/Ошибки/Эйсы) | Атака(Всего/Ошибки/Уд.в блок/Очки) | Блок(Всего/Ошибки/Очки) | Защита(Всего/Ошибки/Отличный) | Приём(Всего/Ошибки/Отличный)

`attackErrors` = невынужденные (quality !== `attack_blocked`), `attackBlocked` = quality === `attack_blocked`.

### API эндпоинты
- `GET /api/health` — healthcheck
- `GET/POST/PUT/DELETE /api/teams[/:id]` — CRUD команд
- `GET/POST/PUT/DELETE /api/matches[/:id]` — CRUD матчей
- `GET/POST /api/matches/:matchId/events` — события матча
- `DELETE /api/matches/:matchId/events/:eventId` — удаление события
- `GET/POST /api/matches/:matchId/special-events` — спец. события
- `GET /api/sync` — pull всех данных (teams, matches, events, specialEvents, rallies)
- `POST /api/sync` — bulk upsert (teams, matches, events, specialEvents, rallies)

### Файлы API
- `src/index.ts` — entry point, Hono app, CORS, маршруты
- `src/db/index.ts` — SQLite инициализация + CRUD-функции + bulkSync (включая rallies)
- `src/db/schema.sql` — DDL: таблицы `teams`, `matches`, `match_events`, `special_events`, `rallies`. Вложенные массивы хранятся как JSON TEXT.
- `src/routes/teams.ts`, `matches.ts`, `events.ts`, `sync.ts` — Hono роуты

### SQLite-схема

5 таблиц:
- `teams` — id, name, short_name, players (JSON), created_at, updated_at
- `matches` — id + все поля Match, lineups и sets как JSON; колонки `set_lineups`, `serving_team_id`, `first_serve_team_id`, `home_libero_id`, `away_libero_id`, `home_libero_replaced_player_id`, `away_libero_replaced_player_id`
- `match_events` — id, match_id, rally_id, set_number, timestamp, team_id, player_id, action, result, score_home, score_away, meta (JSON)
- `special_events` — id, match_id, set_number, timestamp, team_id, type, meta (JSON)
- `rallies` — id, match_id, set_number, rally_number, serving_team_id, server_player_id, score_home_before/after, score_away_before/after, point_won_by_team_id, was_ace, was_serve_error, was_reception_error, timestamp

Синхронизация: offline-first, клиент пишет в IndexedDB → фоновый sync на `/api/sync`. Стратегия: timestamp-merge (по updatedAt). Клиент: `pullFromServer` сравнивает `updatedAt` для teams/matches, берёт новее. Сервер: `upsertTeam/upsertMatch` — `ON CONFLICT DO UPDATE ... WHERE excluded.updated_at > table.updated_at`. Events/SpecialEvents/Rallies — immutable (insert-only, `ON CONFLICT DO NOTHING`).

## Подсказки для агентов

### Частые ошибки при работе с проектом

1. **Не забывать `pnpm build` перед деплоем** — Turborepo кэширует, но `shared` должен быть скомпилирован первым.
2. **Service Worker кэш** — после деплоя пользователь может не видеть изменений. Рекомендовать Ctrl+Shift+R.
3. **`@volleystats/shared` на сервере** — НЕ должен быть в зависимостях серверного package.json. Типы используются только при компиляции.
4. **SQLite миграции** — `ALTER TABLE ADD COLUMN` не поддерживает `IF NOT EXISTS`. Нужно оборачивать в try/catch или проверять наличие колонки.
5. **`autoGenerated` события** — больше не создаются (с момента перехода на ретроактивные обновления), но могут оставаться в старых данных. Всегда фильтровать: `!e.meta?.autoGenerated`.
6. **Соперник с генерическими игроками** — `awayLineup` содержит `away-player-1..6`. ID проверяются через `isAwayPlayerId()`. Display-имена через `getAwayPlayerDisplayName()`.
7. **Ротация при side-out** — применяется к команде, которая ПОЛУЧАЕТ подачу (не к текущему подающему). После ротации вызывается `autoSwapLiberoOut()` для проверки позиции либеро.
8. **Подача соперника** — всегда `result: 'neutral'` (кроме ошибки). Quality обновляется ретроактивно после приёма. Эйс не выбирается вручную — определяется автоматически, когда наш приём = ошибка. Старые `autoGenerated` ace-события (до этого изменения) фильтруются через `!e.meta?.autoGenerated`.
9. **Quality на ошибках** — все error-действия должны иметь quality-тег (`attack_error`, `attack_blocked`, `block_error`, `serve_error`, `reception_error`, `defense_error`, `setting_error`). Без quality ошибка отображается как просто "Ошибка".
10. **Зонно-зависимые кнопки** — `getAvailableActions()` и `getAwayAvailableActions()` принимают `zone?: CourtZone`. Передняя линия (2,3,4) получает доп. блок-опции (soft, touch). `ActionPopover` принимает `zone` и передаёт в engine.

### Архитектурные решения

- **`matchEngine.ts` — чистые функции**, без побочных эффектов и без доступа к БД. Вся персистенция в `matchStore.ts`.
- **`matchStore.ts` — оркестратор**: вызывает engine, сохраняет в Dexie, обновляет состояние Zustand.
- **`live.tsx` — UI-слой**: читает стор, рендерит компоненты, обрабатывает нажатия. Содержит `SubstitutionModal` и хелперы замен (`getSubstitutionHistory`, `getEligibleBenchPlayers`).
- **Либеро** управляется вручную через `LiberoControl` (кнопки «На площадку» / «Убрать»). Поддержка до 2 либеро, но **только один на площадке**. При входе запоминается `homeLiberoReplacements[liberoId] = replacedPlayerId`, при выходе — восстанавливается. Автозамена при ротации: `autoSwapLiberoOut()` проверяет, не оказался ли либеро на передней линии (зоны 2,3,4) или в зоне 1 при своей подаче. `getLiberoEligiblePlayers(lineup, isTeamServing)` — зона 1 доступна для замены только при подаче **соперника**.
- **Замены** — `SubstitutionModal` (двухшаговый: «Кого снять?» → «Кого выпустить?»). Правило обратной замены вычисляется из `specialEvents` текущего сета. `handleSubstitution` обновляет lineup, инкрементирует `homeSubstitutions`, создаёт `SpecialEvent { type: 'substitution', meta: { playerIn, playerOut } }`. Макс 6 замен на сет.
- **Откат розыгрыша** — `undoLastRally()` в matchStore: удаляет последний rally и его события из DB, восстанавливает счёт из `scoreHomeBefore/scoreAwayBefore`, при side-out — обратная ротация (`reverseRotateLineup`) + откат авто-замен либеро (по `specialEvents` с `meta.description === 'auto'`). Кнопка undo в live.tsx работает: внутри rally → `undoLastRallyAction`, между розыгрышами → `undoLastRally`.
- **Подсчёт эйсов** — везде через `meta?.quality === 'ace'` (stats.tsx, ScoreBar.tsx, exportProtocol.ts). Работает одинаково для домашних (result='success', quality='ace') и соперника (result='neutral', quality='ace' после ретроактивного обновления).
- **`updatedEvents` в ActionOutcome** — механизм для ретроактивного обновления существующих событий. `processAction` возвращает `updatedEvents[]`, matchStore сохраняет их через `db.matchEvents.put()` и мержит в стейт.
